# Electron + Vite + React in the Monorepo

> Research intensity: deep
> Date: 2026-03-06
> Status: research complete

## Summary

`electron-vite` is the de-facto build tool for Electron applications using Vite. It wraps a single `electron.vite.config.ts` around three separate Vite builds — main process, preload scripts, and renderer — and handles the Node.js / browser context split automatically. For this monorepo the goal is `apps/electron/` that imports `@repo/db`, `@repo/core`, and `@repo/ui`, runs direct PostgreSQL queries in the Electron main process, and exposes typed IPC endpoints to a React + TanStack Router renderer.

## Key Documentation Findings

### What electron-vite provides

- **One config, three builds.** `electron.vite.config.ts` exports `{ main, preload, renderer }`, each a standard Vite config.
- **Output layout:**
  ```
  out/
  ├── main/index.js
  ├── preload/index.mjs   (ESM preloads get .mjs extension)
  └── renderer/index.html
  ```
- **Package.json `main` field** must point to `./out/main/index.js`.
- **Default source discovery:** `src/main/{index|main}.{ts,js}`, `src/preload/{index|preload}.{ts,js}`, `src/renderer/index.html`.
- **Dev mode** (`electron-vite dev`): starts the Vite dev server for the renderer, then launches Electron — no separate `concurrently` dance needed.
- **HMR** works for the renderer. Main/preload get hot-reloaded (process restart), not HMR.
- **Env prefixes:** `MAIN_VITE_*` for main, `PRELOAD_VITE_*` for preload, `RENDERER_VITE_*` for renderer, `VITE_*` for all three.

### ESM status (important)

Electron has supported ESM in the main process since Electron 28 (Node loader). electron-vite 2.x is itself ESM and supports ESM output. However:

- **Preload scripts ignore `"type": "module"`** from `package.json`. ESM preloads must use the `.mjs` extension — electron-vite handles this automatically when you have `"type": "module"` in the Electron app's `package.json`.
- **Sandboxed preloads cannot use ESM at all.** They must be CJS or fully bundled. The practical solution is to **disable sandbox for the preload** (`sandbox: false` on `BrowserWindow.webPreferences`) while keeping `contextIsolation: true` — this is what the `@electron-toolkit/preload` helper assumes.
- Main process ESM: modules load asynchronously, so call anything that touches Electron APIs (e.g. `app.setPath`) only after `await app.whenReady()`.

### Dependency handling

By default electron-vite externalises everything listed in `dependencies`. Workspace packages (`@repo/*`) **are not** in `node_modules` the same way — Vite detects they are linked and treats them as source code (bundled inline). This is the right behaviour for main process and preload: you want `@repo/core` and `@repo/db` bundled in. Third-party runtime deps like `pg` stay external so electron-builder can include them in the asar.

Control with:

```ts
build: {
  externalizeDeps: {
    exclude: ['@repo/core', '@repo/db'],   // force bundle (usually automatic for workspace links)
    include: ['some-esm-only-pkg'],        // force external
  }
}
```

Or to fully bundle everything (slower, larger, but simpler for packaging):

```ts
build: {
  externalizeDeps: false;
}
```

## Best Practices

### Project structure inside `apps/electron/`

```
apps/electron/
├── electron.vite.config.ts      # three-section build config
├── electron-builder.yml         # packaging config
├── package.json                 # "main": "./out/main/index.js", "type": "module"
├── tsconfig.json                # references tsconfig.node.json + tsconfig.web.json
├── tsconfig.node.json           # for main + preload (node target, no DOM)
├── tsconfig.web.json            # for renderer (dom lib, jsx)
└── src/
    ├── main/
    │   ├── index.ts             # BrowserWindow creation, IPC handlers, DB init
    │   └── ipc/
    │       └── handlers.ts      # all ipcMain.handle calls
    ├── preload/
    │   └── index.ts             # contextBridge.exposeInMainWorld
    └── renderer/
        ├── index.html           # Vite entry HTML
        └── src/
            ├── main.tsx         # React root
            ├── routeTree.gen.ts # generated by TanStack Router plugin
            ├── routes/          # file-based routes
            └── types/
                └── electron.d.ts  # Window.api type declaration
```

### Security: never expose `ipcRenderer` directly

```ts
// preload/index.ts — CORRECT
import { contextBridge, ipcRenderer } from "electron";

contextBridge.exposeInMainWorld("api", {
  getTasks: (projectId: number) => ipcRenderer.invoke("tasks:list", projectId),
  createTask: (input: CreateTaskInput) =>
    ipcRenderer.invoke("tasks:create", input),
});
```

```ts
// preload/index.ts — WRONG (exposes entire ipcRenderer)
contextBridge.exposeInMainWorld("electron", { ipcRenderer });
```

### Main process DB initialisation

```ts
// main/index.ts
import { app, BrowserWindow, ipcMain } from "electron";

import { db } from "@repo/db/client"; // bundled inline by electron-vite

import { registerHandlers } from "./ipc/handlers.js";

app.whenReady().then(async () => {
  // DB connection via @repo/db is available — pg connects lazily on first query
  registerHandlers(ipcMain, db);
  createWindow();
});
```

### IPC handler pattern

```ts
// main/ipc/handlers.ts
import type { IpcMain } from "electron";

import type { Db } from "@repo/db/client";
import { taskService } from "@repo/core";

export function registerHandlers(ipcMain: IpcMain, db: Db) {
  ipcMain.handle("tasks:list", (_event, projectId: number) =>
    taskService.list(db, projectId),
  );
  ipcMain.handle(
    "tasks:create",
    (_event, input: unknown) => taskService.create(db, input), // validate with zod inside the service
  );
}
```

### Renderer type declarations

```ts
// renderer/src/types/electron.d.ts
import type { CreateTaskInput, Task } from "@repo/core";

declare global {
  interface Window {
    api: {
      getTasks: (projectId: number) => Promise<Task[]>;
      createTask: (input: CreateTaskInput) => Promise<Task>;
    };
  }
}
```

### TanStack Router in Electron

Do **not** use `createBrowserHistory` — it requires a server to handle navigation. Use hash history:

```ts
// renderer/src/main.tsx
import { createHashHistory, createRouter } from "@tanstack/react-router";

import { routeTree } from "./routeTree.gen.js";

const router = createRouter({
  routeTree,
  history: createHashHistory(),
});
```

The `@tanstack/router-plugin/vite` plugin works the same as in `apps/react/`.

## Common Pitfalls

### 1. Preload sandboxing breaks Node APIs

From Electron 20+, preload scripts are sandboxed by default. `require('electron')` still works, but Node built-ins and npm packages do not. Two options:

- Set `sandbox: false` in `webPreferences` (required if you want `@electron-toolkit/preload` helpers or any Node import in the preload). Keep `contextIsolation: true`.
- Fully bundle all preload dependencies with `build.externalizeDeps: false` for the preload section.

Recommendation: use `sandbox: false` + `contextIsolation: true` for this project because the preload needs to import from `electron` properly.

### 2. `pg` / `pg-native` and ASAR

`pg` (node-postgres) is a pure-JS package — no native `.node` binary — so it does **not** need `asarUnpack`. However, if you ever add `pg-native` (the C++ binding for better performance), it produces a `.node` file that cannot run from inside an ASAR archive. electron-builder auto-detects `.node` files, but you can be explicit:

```yaml
# electron-builder.yml
asarUnpack:
  - "**/*.node"
```

`pg` itself should be listed under `dependencies` (not devDependencies) in `apps/electron/package.json` so electron-builder includes it in the packaged app.

### 3. pnpm symlinks and electron-builder

electron-builder doesn't always follow pnpm's symlinked `node_modules` correctly. The documented workaround is to add `.npmrc` with `shamefully-hoist=true` in `apps/electron/`, reinstall, and repackage. Alternatively, set `node-linker=hoisted` in `.npmrc` only for the electron app subdirectory.

### 4. Hash routing required in production

`createBrowserHistory` fails in production Electron (file:// protocol has no server). Always use `createHashHistory` in the renderer — the URL will be `file:///path/to/app#/your/route`.

### 5. `DATABASE_URL` must come from Electron, not `.env` at build time

The main process runs on the user's machine. DB connection strings must be stored in app config (e.g. `electron-store`, a settings file, or bundled from a local `.env` loaded at runtime via `dotenv`). Do **not** bake `DATABASE_URL` into the renderer bundle.

### 6. `moduleResolution: Bundler` vs electron-vite

The workspace base tsconfig uses `moduleResolution: Bundler`. For the electron main/preload tsconfigs, you should use `moduleResolution: Node16` or `Bundler` — both work with electron-vite since it uses Rollup under the hood and does not rely on tsc for emit. The renderer tsconfig can reuse `Bundler` as usual.

### 7. Vite and CJS conflicts in main process

If a workspace package (e.g. `@repo/db` which uses `pg`) triggers a CJS/ESM conflict during bundling, add it to `build.externalizeDeps.exclude` explicitly to force bundling, or ensure the package is marked `"type": "module"` (which `@repo/db` already is).

### `electron.vite.config.ts`

```ts
import path from "node:path";
import tailwindcss from "@tailwindcss/vite";
import { tanstackRouter } from "@tanstack/router-plugin/vite";
import react from "@vitejs/plugin-react";
import { defineConfig, externalizeDepsPlugin } from "electron-vite";

export default defineConfig({
  main: {
    plugins: [
      // Externalise true third-party deps (pg, etc.); workspace packages
      // (@repo/*) are linked and auto-treated as source by Vite.
      externalizeDepsPlugin({ exclude: ["@repo/core", "@repo/db"] }),
    ],
    resolve: {
      alias: { "~": path.resolve(__dirname, "src/main") },
    },
  },
  preload: {
    plugins: [
      // Preload must be fully bundled when sandbox: false
      externalizeDepsPlugin({ exclude: ["@repo/core", "@repo/db"] }),
    ],
  },
  renderer: {
    root: "src/renderer",
    plugins: [
      tailwindcss(),
      tanstackRouter({
        target: "react",
        autoCodeSplitting: true,
        routesDirectory: "./src/renderer/src/routes",
        generatedRouteTree: "./src/renderer/src/routeTree.gen.ts",
      }),
      react(),
    ],
    resolve: {
      alias: { "~": path.resolve(__dirname, "src/renderer/src") },
    },
  },
});
```

### TypeScript config split

The main/preload and renderer need separate tsconfigs because they target different environments:

```json
// tsconfig.node.json — for main + preload
{
  "extends": "@repo/tsconfig/base.json",
  "compilerOptions": {
    "lib": ["ES2022"],
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "types": ["node"],
    "noEmit": false,
    "emitDeclarationOnly": true
  },
  "include": ["src/main/**/*", "src/preload/**/*", "electron.vite.config.ts"]
}
```

```json
// tsconfig.web.json — for renderer
{
  "extends": "@repo/tsconfig/base.json",
  "compilerOptions": {
    "lib": ["ES2022", "DOM", "DOM.Iterable"],
    "jsx": "react-jsx",
    "types": [],
    "paths": {
      "~/*": ["./src/renderer/src/*"]
    }
  },
  "include": ["src/renderer/**/*"]
}
```

### Turborepo integration

Add to `turbo.json` global tasks — electron's `dev` task needs `@repo/db` and `@repo/core` built first:

```json
// The existing turbo.json "build" pipeline already handles ^build dependency
// No changes needed — turbo will build @repo/db and @repo/core before @repo/electron
```

The `package.json` workspace dep on `@repo/core` and `@repo/db` is sufficient for Turborepo to infer the build order.

## Implementation Hints

### Main process window creation (complete minimal example)

```ts
// src/main/index.ts
import path from "node:path";
import { is } from "@electron-toolkit/utils";
import { app, BrowserWindow, ipcMain, shell } from "electron";

import { db } from "@repo/db/client";

import { registerHandlers } from "./ipc/handlers.js";

function createWindow(): void {
  const win = new BrowserWindow({
    width: 1200,
    height: 800,
    webPreferences: {
      preload: path.join(__dirname, "../preload/index.mjs"),
      sandbox: false, // required: lets preload use Node/Electron APIs
      contextIsolation: true, // required: enforces security boundary
    },
  });

  win.webContents.setWindowOpenHandler(({ url }) => {
    shell.openExternal(url);
    return { action: "deny" };
  });

  if (is.dev && process.env.ELECTRON_RENDERER_URL) {
    void win.loadURL(process.env.ELECTRON_RENDERER_URL);
  } else {
    void win.loadFile(path.join(__dirname, "../renderer/index.html"));
  }
}

app.whenReady().then(() => {
  registerHandlers(ipcMain, db);
  createWindow();

  app.on("activate", () => {
    if (BrowserWindow.getAllWindows().length === 0) createWindow();
  });
});

app.on("window-all-closed", () => {
  if (process.platform !== "darwin") app.quit();
});
```

### Preload with typed surface

```ts
// src/preload/index.ts
import { contextBridge, ipcRenderer } from "electron";

import type { CreateTaskInput, Task } from "@repo/core";

const api = {
  tasks: {
    list: (projectId: number): Promise<Task[]> =>
      ipcRenderer.invoke("tasks:list", projectId),
    create: (input: CreateTaskInput): Promise<Task> =>
      ipcRenderer.invoke("tasks:create", input),
  },
};

contextBridge.exposeInMainWorld("api", api);

// Re-export type so renderer can import it
export type ElectronAPI = typeof api;
```

```ts
// src/renderer/src/types/electron.d.ts
import type { ElectronAPI } from "../../../preload/index.js";

declare global {
  interface Window {
    api: ElectronAPI;
  }
}
```

### Renderer entry with hash routing

```ts
// src/renderer/src/main.tsx
import { StrictMode } from 'react'
import { createRoot } from 'react-dom/client'
import { RouterProvider, createHashHistory, createRouter } from '@tanstack/react-router'
import { routeTree } from './routeTree.gen.js'
import './styles/globals.css'

const router = createRouter({
  routeTree,
  history: createHashHistory(),
})

declare module '@tanstack/react-router' {
  interface Register {
    router: typeof router
  }
}

const root = document.getElementById('root')
if (!root) throw new Error('Root element not found')
createRoot(root).render(
  <StrictMode>
    <RouterProvider router={router} />
  </StrictMode>,
)
```

### Reading DATABASE_URL at runtime in the main process

Since users run this on their machines, load the connection string from app config or a local `.env` beside the installed app:

```ts
// src/main/index.ts — dev: load from repo root .env; prod: from app userData
import path from "node:path";
import { is } from "@electron-toolkit/utils";
import dotenv from "dotenv";
import { app } from "electron";

// Now import db — it will pick up DATABASE_URL from process.env
import { db } from "@repo/db/client";

if (is.dev) {
  dotenv.config({ path: path.resolve(__dirname, "../../../.env") });
} else {
  dotenv.config({ path: path.join(app.getPath("userData"), ".env") });
}
```

## Skill Recommendation

**Recommendation:** Yes

**Reasoning:** The electron-vite + monorepo combination has multiple non-obvious interaction points — the ESM preload `.mjs` requirement, the sandbox/contextIsolation pairing, the dependency externalization strategy for workspace packages, the mandatory `createHashHistory` for TanStack Router, and the `DATABASE_URL` runtime-vs-build-time split. These are exactly the kinds of footguns that trip up fresh implementations and that a skill can reliably steer around.

**Suggested skill scope:** A skill covering `apps/electron/` scaffolding in this specific monorepo — the exact file structure, the `electron.vite.config.ts` template with correct `externalizeDepsPlugin` usage, the preload/contextBridge/IPC handler pattern, the TanStack Router hash history setup, the split tsconfig approach, and the `DATABASE_URL` runtime loading pattern. Should trigger when the user asks to add/modify Electron functionality or IPC channels.
